# dev-workgraph MVP Requirements

## Goal

The MVP goal is to check whether dev-workgraph can reconstruct useful engineering context from Git history.

The MVP should answer the following questions:

1. Can we export commits and patches in a stable, reproducible format?
2. Can a local model generate useful commit-level descriptions?
3. Can we build an immutable graph from commits, patches, summaries, files, and project areas?
4. Can the resulting graph show meaningful work context and help decide the next direction?

This MVP is not a full resume generator, portfolio builder, or final interview assistant.
It is an evaluation prototype for the core idea.

⸻

## 1. Commit and patch export

The system must be able to export commits from a Git repository.

For MVP, export can be manual or semi-automated.

Each exported commit must be stored in a folder named by commit Unix timestamp:

```
~/.workgraph/data/commits/[unix-timestamp]/[hash].patch
```
Example:
```
~/.workgraph/commits/1717428123/b0648088.patch
~/.workgraph/commits/1717428123/b0648088.json
```

The patch must be generated from Git, if possible, using a reproducible command.

Suggested command:
```
git show --format=fuller --find-renames <commit-hash> > ~/.workgraph/data/commits/<unix-timestamp>/<commit-hash>.patch
```
The export should preserve:

* commit hash
* commit date
* author
* commit title/message
* changed files
* patch content

The patch export should exclude obvious noise where possible:
```
node_modules/**
dist/**
build/**
target/**
coverage/**
.next/**
*.min.js
*.map
package-lock.json
yarn.lock
pnpm-lock.yaml
```
For MVP, noise filtering can be implemented either during patch export or during patch analysis.

⸻

## 2. Commit JSON summary

For every exported patch, the system must create a JSON file next to the patch:
```
~/.workgraph/data/commits/[unix-timestamp]/[hash].json
```
Example:
```
~/.workgraph/data/commits/1717428123/b0648088.json
```
The JSON file must describe what was done in the commit.

The summary should be generated by a local model for MVP, if possible.

### Required JSON fields:

```json
{
  "commitHash": "...",
  "timestamp": 0,
  "title": "...",
  "summary": "...",
  "changedFiles": {
    "added": [],
    "deleted": [],
    "modified": [],
    "renamed": []
  },
  "importantFolders": [],
  "careerSignal": 0.0,
  "technicalSignal": 0.0,
  "architectureSignal": 0.0,
  "securitySignal": 0.0,
  "questions": [],
  "confidence": "low | medium | high"
}
```

#### Field meaning

`summary` should explain what changed in plain language.

`changedFiles` should contain all files touched by the commit.

`importantFolders` should contain folders that appear important for understanding the change.

`changeTypes` may include values such as:
```
feature
bugfix
refactoring
security
infrastructure
testing
configuration
developer-tooling
architecture
documentation
```

`careerSignal` should estimate whether this commit may be useful for professional storytelling.

`technicalSignal` should estimate technical depth.

`architectureSignal` should estimate whether the change affects structure, boundaries, modules, or system design.

`securitySignal` should estimate whether the change relates to authentication, authorization, identity, data protection, permissions, or platform security.

#### Example:

```json
please create
```

### 3. Import data into database and build graph

The system must import exported commit data into a database.

For MVP, SQLite is enough.

The system must create graph nodes and edges from:

* commits
* patches
* commit summaries
* changed files
* project folders / project areas
* generated context nodes

The graph must be append-only where possible.

Existing summaries should not be overwritten.
New analysis should create new nodes or new versions.

⸻

### 4. MVP graph model

The MVP graph should include at least the following node types:
```
repository
commit
patch
commit-summary
file
folder-context
area-context
question
```
Optional node types:
```
group-summary
architecture-context
evidence-review
user-note
```

Required edge types
```
COMMIT_HAS_PATCH
COMMIT_HAS_SUMMARY
COMMIT_TOUCHES_FILE
FILE_LOCATED_IN_FOLDER
FOLDER_PART_OF_REPOSITORY
SUMMARY_DERIVED_FROM_PATCH
SUMMARY_RAISES_QUESTION
SUMMARY_RELATES_TO_AREA
AREA_CONTEXT_UPDATED_BY_SUMMARY
SUPERSEDES
```
⸻

### 5. Project folder and area context

Project folders must be analyzed as a separate context layer.

The purpose of folder context is not to prove what was done in a commit.

The purpose is to explain:

* what part of the project was affected
* what subsystem the files belong to
* why this area may exist
* what architectural role this area may have
* what questions should be asked to confirm the rationale

For each commit, the system should detect affected folders and update or create related context nodes.

Example:

keycloak-plugins/Manufacturer-Plugin

May produce or update an area context node:
```
{
  "commitHash": "b0648088",
  "timestamp": 1717428123,
  "title": "Production build and deployment automation",
  "summary": "Added production-oriented build and deployment automation, including Maven build profiles, ZIP assembly, backend package metadata, build and deploy scripts, static asset packaging, and an HTTP upgrade endpoint.",
  "changedFiles": {
    "added": [
      "assembly/production.xml",
      "backend/build.sh",
      "backend/deploy.sh",
      "backend/package.json",
      "backend/pom.xml"
    ],
    "deleted": [],
    "modified": [
      ".gitignore",
      "backend/server.js"
    ],
    "renamed": []
  },
  "importantFolders": [
    "assembly",
    "backend"
  ],
  "changeTypes": [
    "infrastructure",
    "deployment",
    "developer-tooling"
  ],
  "careerSignal": 0.72,
  "technicalSignal": 0.68,
  "architectureSignal": 0.55,
  "securitySignal": 0.1,
  "questions": [
    "Was this used for production deployment?",
    "Was this replacing a manual deployment process?",
    "Was this owned by you or part of a team effort?"
  ],
  "confidence": "medium"
}
```

Area context must be treated as contextual information, not as direct proof of user impact.

### 6. Incremental graph construction

The graph should be built incrementally in chronological order.

Commits should be processed from oldest to newest.

For every commit:

1. Read patch.
2. Read or generate commit JSON summary.
3. Create commit node.
4. Create patch node.
5. Create commit-summary node.
6. Create file nodes if missing.
7. Create or update folder-context nodes.
8. Create or update area-context nodes.
9. Create edges between all related nodes.
10. Generate questions if the commit or area looks important.

When a new commit affects an existing project area, the system should not overwrite the existing area context.
Instead, it should create a new version of the context node and connect it with SUPERSEDES.

Example:
```
area-context:keycloak-plugins:v1
area-context:keycloak-plugins:v2
area-context:keycloak-plugins:v2 --SUPERSEDES--> area-context:keycloak-plugins:v1
```
⸻

### 7. Questions and missing context

At every analysis stage, the system should generate questions when important context is missing.

Questions may be attached to:

* commit summaries
* folder-context nodes
* area-context nodes
* future group summaries

Examples:

Why was this module introduced?
Was this change used in production?
Was this related to customer requirements?
Was this part of a security boundary?
Was this replacing a manual process?
Did this affect multiple tenants or realms?
Was this your own design, implementation, or maintenance work?

Questions are important because Git history alone usually cannot explain business context, ownership, or impact.

⸻

### 8. Graph usefulness analysis

After importing and processing the data, the MVP must analyze whether the graph is useful.

The system should produce a report with:

1. Number of commits processed.
2. Number of generated summaries.
3. Number of project areas detected.
4. Top areas by commit activity.
5. Top areas by architecture/security signal.
6. Top commits by career signal.
7. Most common generated questions.
8. Potential workstream candidates.
9. Weaknesses or missing context.
10. Recommendation for next development step.

Example output:

Processed commits: 300
Commit summaries: 300
Project areas detected: 18
High-signal areas:
- Keycloak plugins
- Keycloak configuration
- AWS delivery
- Integration tests
  Potential workstreams:
- Identity and access platform hardening
- Manufacturer realm management
- Device and email verification
- AWS edge delivery automation
- Integration testing framework
  Main missing context:
- production usage
- user ownership
- customer/business impact
- reason for module boundaries

⸻

### 9. MVP success criteria

The MVP is successful if the resulting graph can show useful engineering context from one real repository.

Minimum success criteria:

- 100+ commits can be exported and processed
- each commit has a JSON summary
- graph nodes and edges are created successfully
- project areas are detected
- area context evolves over time
- useful questions are generated
- top areas and high-signal commits look meaningful

Strong success criteria:

- the graph reveals forgotten work
- generated summaries are mostly accurate
- high-signal areas match real important subsystems
- questions help recover missing context
- at least 5 potential achievement/workstream candidates appear useful

Failure criteria:

- summaries are too generic
- area context is mostly wrong
- graph does not reveal anything useful
- too much noise from generated files or folder structure
- local model cannot summarize patches reliably

⸻

### 10. Out of scope for MVP

The following features are out of scope for MVP:
```
resume generation
PDF/DOCX export
public portfolio generation
multi-user support
cloud synchronization
full UI
advanced vector search
complex graph database
automatic Jira/GitHub integration
production-ready plugin system
```
These may be added later if the graph proves useful.

### 12. Core principle

The MVP must preserve the distinction between evidence and context.

commits and patches = evidence
summaries = derived interpretation
folder and area context = architectural explanation
questions = missing human context
graph = relationships between all of them

The system should not overclaim.

It should help reconstruct what happened, why it may have mattered, and what still needs human confirmation.
